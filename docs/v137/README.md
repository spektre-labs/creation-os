# v137 σ-Compile — AOT compilation of the σ-decision tree

Source: `src/v137/compile.[ch]` · `src/v137/main.c`
Gate:   `benchmarks/v137/check_v137_compile_llvm_smoke.sh`
Build:  `make creation_os_v137_compile` · `make check-v137`

## Problem

The σ-layer touches **every token**. Today's v60..v100 gate is
interpreted: a table of thresholds loaded from `config.toml` at
start-up, a function-pointer dispatch per layer, a compare-and-OR
reduction per call. Modern `-O3` already auto-vectorises that loop
into SIMD compares, but the table load + dispatch cost remain.

## σ-innovation

v137 emits a **per-profile, branchless, constant-folded** σ-gate
as C source that the local toolchain lowers natively:

```c
/* Generated by v137 σ-compile.  Do not edit by hand. */
#include <stdint.h>

int cos_v137_compiled_gate(const float ch[8]) {
    return
        (ch[0] > 0.500000f) |
        (ch[1] > 0.500000f) |
        (ch[2] > 0.500000f) |
        (ch[3] > 0.500000f) |
        (ch[4] > 0.500000f) |
        (ch[5] > 0.500000f) |
        (ch[6] > 0.500000f) |
        (ch[7] > 0.500000f);
}
```

Every threshold is a C literal. No table loads. `$(CC) -O3` lowers
this to whatever IR the toolchain prefers (LLVM on clang, GIMPLE
on gcc). v137.1 emits LLVM IR directly via `clang -S -emit-llvm`
and links it; v137.0 trusts the toolchain.

## Three surfaces

| Function | Role |
|----------|------|
| `cos_v137_emit_source(tau, path)` | generate a specialised `.c` for a given profile |
| `cos_v137_interpreted_gate(tau, ch)` | runtime reference (what v60..v100 does today) |
| `cos_v137_example_compiled_gate(ch)` | hand-written embedded "compiled" gate specialised for τ=0.50 — lets v137 run end-to-end from a single binary without invoking `$(CC)` at run time |

## Merge-gate measurements

| Claim | Check |
|-------|-------|
| Interpreted ≡ compiled on 8 edge-case vectors | `check-v137` |
| `--emit` produces C that compiles clean under `$(CC) -O3 -c` | `check-v137` |
| `--emit --tau 0.1,0.2,…,0.8` bakes those literals into the output | `check-v137` |
| Both paths < **500 ns/call** on the merge-gate host | `check-v137` |

On an M3 (Apple Silicon) both paths measure **≈ 0.6–1.2 ns/call** —
roughly 400× under the spec ceiling.

## Honest non-claim

v137.0 does **not** assert a *relative* speedup over the
interpreted path inside a single translation unit. Modern `-O3`
auto-vectorises the 8-element runtime loop into SIMD compares, so
on an M3 the interpreted path is already near the hardware floor.

The AOT win materialises *across function boundaries*: no
function-pointer dispatch, no τ-table prefetch, constant folding
into callers. That full setting is v137.1's job. v137.0 ships the
generator + the compilation smoke test + the absolute-ceiling
latency bound so the pipeline is ready for v137.1 to plug in.
